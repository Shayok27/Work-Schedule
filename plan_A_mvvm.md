The goal of this plan is to implement the registration page by following the MVVM architecture. In this structure, index.html is used as the View, main.js works as the View Controller, authViewModel.js acts as the ViewModel, and firebaseService.js serves as the Model. This approach helps separate the user interface, application logic, and data management into different files, making the project easier to understand and maintain.

The index.html file is responsible only for displaying the registration page interface. It contains the form fields for Full Name, Email, Password, and Confirm Password. It also includes areas where error or success messages can be shown to the user. No validation or Firebase-related code should be written inside this file because its purpose is only to handle the visual part of the application.

The main.js file works as the connection between the user interface and the application logic. It listens for events such as the user clicking the Sign Up button or submitting the form. When the form is submitted, it collects the entered information and sends it to the ViewModel. After receiving a response, it updates the interface by showing either a success message or an error message. If the registration is successful, it can also redirect the user back to the login page.

The authViewModel.js file contains the main registration and login logic. It checks whether all required fields are filled in, verifies that the email is in the correct format, ensures that the password contains at least six characters, and confirms that the password and confirm password fields match. If any validation fails, it returns an error message. If all validation checks pass, it sends the data to the Firebase service and returns a structured result such as a success or failure message.

The firebaseService.js file handles all communication with Firebase. It initializes Firebase and provides functions for creating a new account, saving the user profile, logging in, and retrieving the user role. When a new user registers, this file creates the account in Firebase Authentication and then stores the user’s information in Firestore. The stored information includes the user’s name, email, date of creation, and a default role such as “Pending,” which can later be changed by the administrator.

The implementation process begins by building the registration form in index.html. The form should have the ID signupForm and contain input fields for the user’s name, email, password, and confirm password. It should also include elements that display error and success messages.

Next, event handling is added in main.js. A submit event listener is attached to the registration form. When the form is submitted, the input values are collected and passed to the signup() method in authViewModel.js. Depending on the returned result, the page displays an error message or a success message. If the registration is successful, the user is redirected to the login page.

The validation logic is then implemented in authViewModel.js. The signup() function should return an object such as { success: true, message: "Registration complete" } if everything is valid. If there is a problem, it should return an object such as { success: false, message: "Invalid email" }. Only after successful validation should the ViewModel call the Firebase functions.

Finally, several edge cases should be tested to make sure the registration page works correctly. These include leaving fields empty, entering an invalid email, using a weak password, entering passwords that do not match, attempting to register with an email that already exists, losing the internet connection, or failing to save data to Firestore.
